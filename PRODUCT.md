# Product

## Register

brand

## Users

Developers, both the author and peers, studying how chunked and resumable file uploads work. They arrive from a portfolio link or a GitHub repo, expecting a working demo they can poke at and source code they can read. They care about correctness, clarity, and whether the person who built this actually understands the engineering underneath.

## Product Purpose

A single-page demo of chunked and resumable file uploads built with Vanilla JS, Node.js, Express, and Multer. It exists to prove two things at once: that the author understands the concept deeply enough to implement it from scratch, and that the implementation is clean enough for another developer to learn from. Success is a viewer thinking "this person knows what they're doing" within seconds of landing on the page.

## Brand Personality

Technical, precise, confident. The interface communicates engineering competence through its structure and attention to detail. Not trying to impress with flourishes; the quality of the work itself is the statement. Every element earns its place.

## Anti-references

- Generic Bootstrap or Tailwind tutorial projects with default blue buttons and unstyled forms. The "I followed a tutorial" look.
- Overly corporate SaaS dashboards with gradient hero metrics, identical card grids, and "Welcome back!" headers.
- Busy, cluttered admin panel templates with sidebars, breadcrumbs, and chrome that overwhelms the actual content.
- Any interface where the styling feels like an afterthought bolted onto working code.

## Design Principles

1. **Show the craft.** Every visible detail, from spacing to state transitions, demonstrates that the author cares about quality beyond "it works."
2. **Teach through structure.** The interface itself should make the chunked upload process legible. A developer watching the UI should understand what is happening at each stage without reading the source.
3. **Confidence through restraint.** Knowing what to leave out matters as much as what to include. No element exists for decoration alone.
4. **Respect the viewer's intelligence.** No hand-holding, no patronizing copy, no unnecessary chrome. Developers are the audience; trust them.

## Accessibility & Inclusion

WCAG AA compliance. Ensure sufficient color contrast ratios (4.5:1 for normal text, 3:1 for large text), visible focus indicators, keyboard navigability for all interactive elements, and appropriate ARIA labels where semantic HTML is insufficient. Support `prefers-reduced-motion` for any animations.
